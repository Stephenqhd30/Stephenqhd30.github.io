控制定时任务的执行

为啥?

  1. 浪费资源，想象 10000 台服务器同时“打鸣”
  2. 脏数据，不如重复插入

要控制定时任务在同一时间只有一个服务器能执行

怎么做？

  1. 分离定时任务程序和主程序，只在一个服务器运行定时任务。成本太大

  2. 写死配置，每个服务器都执行定时任务，但是只有ip符合配置的服务器才真实执行业务逻辑，其他的直接返回。成本最低，但是我们的IP可能是不固定的，把IP写的太死了

  3. 动态配置，配置是可以是很轻松的、很方便的更新的，但是只有 ip 符合配置的服务器才能真实的执行业务逻辑

    • Redis
    • 数据库
    • 配置中心（Nacos、 Apollo、Spring Cloud Config）

    问题： 服务器多了、IP不可控还是很麻烦、还是需要人工修改

  4. ​ 分布式锁，只有强大了锁的服务器才能执行业务逻辑

    单机就会存在单点故障

锁

在有限资源的情况下，控制同一时间（段）只有某些线程（用户 / 服务器）能访问到资源

Java 实现锁：synchronized 关键字，并发相关的包

问题：只对单个JVM有效

分布式锁

为什么需要分布式锁？

  1. 在有限资源的情况下，控制同一时间（段）只有某些线程（用户 / 服务器）能访问到资源
  2. 单个锁只对单个JVM有效

分布式锁实现的关键

抢锁机制

怎么保证同一时间只有 1 个服务器能抢到锁

核心思想就是：先来的人把数据修改成自己的标识（服务器IP），后来的人发现表示已经存在，就抢锁失败，继续等待

等先来的人执行方法结束把表示清空，其它人继续抢锁

  • MySQL数据库： select for update 行级锁（最简单）
  • 乐观锁
  • :heavy_check_mark: Redis 实现存标识 （内存数据， 读取速度快）。支持 setnx，luo脚本，比较方便我们实现分布式锁。

setnx： set if not exists 如果不存在，则设置成功才会返回true，否则返回false

  • ZooKeeper 实现

注意事项：

  1. 用完锁要释放（腾地方）
  2. 锁一定要加过期时间 :heavy_check_mark:
  3. 如果方法执行事件过长，锁提前过期了

  • 问题：

    1. 连锁效应：释放到别人的锁
    2. 这样还是会存在多个方法同时执行的情况

  • 解决方法：续期

  1. 释放锁的时候，有时候先判断出是自己的锁，但是这个时候锁过期了，最后还是释放了别人的锁

Redis + Lua 脚本

  1. Redis如果是集群

拒绝自己写！！！

Redisson 实现分布式锁

Java 客户端，数据网络

实现了很多Java里支持的接口和数据结构

Redisson是一个Java客户端，提供了大量分布式数据集简化对Redis的操作和使用，可以让开发者像使用本地集合一样使用 Redis，完全感知不到 Redis 的存在

2种引入方式

  1. Spring boot stater 引入 （不推荐，版本更新迭代太快，容易冲突）
  2. 直接引入Redisson

  1. waitTime 设置成0， 只抢一次，抢不到就放弃
  2. 注意释放锁要写在finaly中

看门狗机制

Redisson中提供的续期机制

会开启一个监听线程，如果方法还没有执行完，就帮你重置Redis锁的过期时间

原理：

  1. 监听当前线程（没10s续期一次）
  2. 如果线程挂掉（注意debug模式也会被当场服务器宕机）

实现类：Redisson

组队功能

理想的应用场景：
我要跟别人一起参加竞赛或者做项目，可以发起队伍或者加入别人的队伍

  • 随机匹配

需求分析：

  1. 用户可以创建一个队伍，设置队伍的人数、队伍名称（标题、描述）、超时时间
  • 队伍、剩余的人数
  • 聊天？
  • 公开 或者 private 或者 加密
  • 信息流中不展示过期的队伍
  • 根据标搜索队伍
  1. 用户可以加入队伍（其它人、未满、未过期），允许加入多个队伍，但是需要有个上限

是否需要队长同意？筛选？审批？

  1. 用户可以退出队伍（如果队长推出，权限顺位转移给第二早加入的用户）
  2. 队长可以解散队伍
  3. 分享队伍 => 邀请其他用户加入队伍
  4. 队伍人满之后可以发送消息通知
  5. 用户最多创建5个队伍

组队的实践

  • 库表设计
  • 增删改查
  • 业务逻辑开发

数据库表设计

队伍表 team

字段：

  • id 主键 bigint （最简单、连续，放在 url 上比较简短， 但是缺点就是爬虫）
  • name 队伍名称
  • description 描述
  • maxNum 最大人数
  • expireTime 过期时间
  • userId 创建人 id
  • password 密码
  • status 0 - 公开， 1 - 私有， 2 - 加密
  • createTime 创建时间
  • updateTime 更新时间
  • isDelete 是否被删除

建表语句

两个关系：（ 用户和队伍的关系 ）

  1. 用户加入那些队伍？
  2. 队伍有哪些用户

方式：

  1. √ 建立用户 - 队伍关系表 teamId userId
  2. 用户表补充已加入队伍的字段，队伍表补充已加入的用户的字段（不用写多对多的代码，可以直接根据队伍查用户，根据用户查队伍）

用户 - 队伍表 user_team

字段：

  • id 主键
  • userId 用户 id
  • teamId 队伍 id
  • joinTime 加入时间
  • createTime 创建时间
  • updateTime 更新时间
  • isDelete 是否被删除

后端代码实现

为什么需要请求参数包装类？

  1. 请求参数名称 / 类型和实体类不一样
  2. 有一些参数用不到，如果要自动生成接口文档，会增加理解成本
  3. 对个体实体类映射到同一个对象

为什么需要包装类？

  • 可能有些字段需要隐藏，不能返回给前端
  • 或者有些字段某些方法是不关心的
  • 什么是DTO
