编写一次性任务

for循环插入数据的问题：

  1. 奖励和释放数据库链接（批量查询解决）
  2. for循环是绝对线性的（并发）

并发需要注意执行的先后顺序无所谓，不要使用非并发类的集合

数据库慢？预先把数据查出来，放到一个更快读取的地方，不在查询数据库了（缓存）

预加载缓存，定时更新缓存。（定时任务）

多个机器都需要执行任务嘛？（分布式锁：控制同一时间只有一台机器去执行定时任务，其他机器不用重复执行了）

数据查询的慢怎么办？

用缓存：提前把数据读取出来保存好（通常保存到读写更快的介质，比如内存），就可以更快的读写

缓存的实现

  • Redis（分布式缓存）
  • memcached（分布式）
  • Etcd（云原生架构的一个分布式存储，存储配置，扩容能力）
  • ehcache（单机的）
  • 本地缓存（Java内存Map）
  • Caffeine（Java内存缓存， 高性能）
  • Google Guava

Redis

NoSQL数据库

Key-Value 存储系统（区别于MySQL， 他存的是键值对）

Redis的数据结构

String 字符串类型： name：”Stephen”

List 列表类型：names： [“Stephen”, “Popcorn”]

Set集合：names：[“Stephen”, “Popcorn”]（值不能重复）

Hash哈希：nameAge：{“Stephen”: 1, “Popcorn”: 2}

Zset集合：names：{Stephen-9, Popcorn-10}(适合做排行榜之类的)

bloomfilter

geo（计算地理位置）

hyperloglog（pv/uv）

pub/sub（发布订阅，类似消息队列）

Java Spring里的实现方式

  • Spring Data Redis（推荐）

  • Jedis

    • 独立于Spring操作Redis的Java客户端
    • 默认是线程不安全的，要配合Jedis Pool使用

  • Lettuce

    • 高阶的操作 Redis 的 Java 客户端
    • 异步，连接池

  • Redisson

    • 分布式操作 Redis 的 Java 库客户端， 让你想使用本地的集合一样操作Redis

  • JetCache

对比：

  1. 如果使用的是Spring，并且没有过多的定制化要求，可以使用 Spring Data Redis
  2. 如果你用的不是 Spring，并且追求简单没并且没有过高的性能要求，并且可以是用 Jedis + Jedis Pool
  3. 如果你使用的不是 Spring，并且追求高性能，高定制化，可以用Letture，支持异步，连接池
  4. 如果你的项目使用的是分布式，需要使用到一些分布式的特性（比如分布式锁、分布式集合）， 推荐使用Redisson

systemId.moduleId:func（不要和别人设计的有冲突就行）

！！！！Redis内存不能无限增加，一定要设置过期时间

缓存预热

问题： 第一个用户访问还是很慢，也能一定程度上保护数据库

缓存预热的优点：

  1. 解决上面的问题，可以让用户始终访问很快

缺点：

  1. 增加开发的成本（你需要要额外的开发、设计）
  2. 预热的时机和时间如果错了，有可能你缓存的数据不对或者太老
  3. 需要占用额外空间

分析优缺点的时候，需要先打开思路，从整体上从 0 到 1 的链路上去分析

怎么缓存预热
  1. 定时任务
  2. 模拟触发（定时触发）

实现：

用定时任务，每天刷新所有用户的推荐列表

注意点：

  1. 缓存预热的意义（新增少、总用户多）
  2. 缓存的空间不能太大，需要预留给其他的缓存空间
  3. 缓存数据的周期（此处每天一次）
定时任务实现
  1. SpringBoot Scheduler （Spring boot默认整合了）@Sched
  2. Quartz（独立于 Spring 存在的定时任务框架）
  3. XXL-Job 之类的分布式任务调度平台（界面+SDK）

第一种方式

  1. 主类开启@EnableScheduling

  2. 要给定时任务执行的方法添加@Scheduling 注解, 指定 cron 表达式或者执行频率

    • crontab在线转换工具
