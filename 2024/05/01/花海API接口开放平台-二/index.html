API 算法签名认证

本质：

  1. 签发签名
  2. 使用签名（校验签名）

为什么需要？

  1. 保证安全性，不能随便一个人调用

怎么实现：

通过 http request header 头传递参数
  1. accessKey: 调用的表示 userA， userB（复杂、无序、无规律）
  2. secretKey：密钥（复杂、无序、无规律），该参数是不传递的，不放在请求头中
  3. 用户参数
  4. sign
  5. nonce 随机数
  6. timestemp 时间戳

加密方式（对称加密，非对称加密，md5签名（不可解密））
用户参数 + 密钥 => 签名生成算法（Hash） => 不可解密的值

（类似用户的账号和密码， 区别是AK、SK是无状态的）

千万不要把密钥直接在服务器之间传递，有可能会被拦截
  1. 怎么知道这个签名对不对呢？？

服务端用一摸一样的参数和算法去生成签名，只要和用户传的一致，就表示一致。

怎么防止重放
  1. 加 nonce 随机数， 只能用一次
    1. 服务端需要保存用过的随机数
  2. 加 timestamp 时间戳，校验时间戳是否过期

AK/SK的分配

  • 用户在注册的时候，为其分配accessKey， secretKey，使用的是Hutool工具调用MD5生成摘要， 生成的过程中加入了随机数
  • accessKey 用于标识用户
  • secretKey 仅作本地加密使用，不参与网路传输（只存在服务器的数据库里面）
  • 需要补充的是，password和ak sk在数据库中都是以摘要的形式存储

AK/SK的使用

  • 用户是感知不到 accessKey 和 secretKey 的
  • 程序从数据库中查询出来他们对应的ak sk来调用接口

签名的生成

  • 拼接用户的请求的参数（请求参数名和字母升降序排列）和 secretKey，使用散列函数生成签名
  • 补充：为了方式时间戳和随机数被篡改，也可以将时间戳和随机数加入签名的生成中
  • 使用HuTool提供的SHA256生成摘要

签名的校验

  • 根据请求头中的 accessKey，查询出当前调用的用户并且获取该用户的secretKey
  • 使用相同的方法生成签名（拼接用户的请求参数和secretKey使用散列函数生成签名）
  • 比较请求头中的签名和生成的签名是否一致

API签名认证的作用

  1. 身份认证

accessKey 和 secretKey 是唯一分配给用户的一对密钥，如果 accessKey 对应的 secretKey 能够生成合法的签名，说明 accessKey 发起的请求的用户是合法的

  1. 参数伪造

因为签名是由请求参数 body + secretKey 经过散列函数加密之后生成的，如果请求参数被篡改，那么生成的签名和请求头中携带的签名肯定是不一致的，无法噶通过签名的校验，可因此能够防止参数的伪造

  1. 重放攻击

什么是重放攻击

  1. 攻击者通过重新发送之前的请求或者响应数据包，视图绕过系统的安全控制

为什么能防止重放攻击

  1. 使用timestamp+nonce机制防止重放攻击
  2. nonce是一个随机，用来识别没一个被签名的请求，每一个随机数只允许被使用一次（服务器会存储使用过的随机数），弱国有一个请求和之前的某一个请求使用了相同的随机数，那么这个请求是被重放的非法请求
  3. 为了降低服务器维护随机数的成本，引入timestamp。有新的请求进来的时候，首先检查timestamp，如果和当前时间差超过五分钟则拒绝（防止攻击者长时间重放该请求）如果不超过五分钟，则检查nonce，如果nonce已经使用过了，说明该请求是重放的，如果没有使用过，则存入数据库中，并且删除超过五分钟后的timestamp的随机数。（timestamp和nonce也是一对一对存储的）

开发一个简单易用的 SDK

理想情况： 开发者只需要关心调用那些接口，传递哪些参数，就像调用自己写的代码一样简单

开发 stater 的好处：开发者引入了之后，可以直接在 application.yml中写配置项，自动创建客户端，spring-boot-configuration-processor的作用是自动生成配置的代码

Stater 开发流程

  • 初始化项目，环境依赖，移除jar包的build文件
  • mvn install 打包为本地的文件依赖
  • 创建新项目（复用serve项目）、测试
  • 发布stater到maven的中心仓库中

在客户端中配置调用我们自己开发的stater
